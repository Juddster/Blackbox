# Blackbox UI Brainstorming

This document holds the UI concepting branch for Blackbox.

The goal is not to design screens pixel-by-pixel yet. The goal is to identify the major user-facing surfaces, the flow between them, and the product behaviors the UI needs to support.

## Brainstorming Branch 3: UI Concepting

### Core Objective

The UI should make Blackbox feel like:
- a passive recorder that usually just works
- a semantic timeline rather than a raw sensor dashboard
- a system that is honest about uncertainty
- a tool for correcting and curating meaning when needed
- a product that can turn history into something presentable and shareable

The UI should not feel like:
- a confusing diagnostic console
- a manual GIS editing tool
- a fitness tracker that only makes sense during active sessions

### Likely Primary User Surfaces

Given the product definition so far, the app likely needs at least these main surfaces:
- timeline
- segment detail
- review queue
- collection view
- export flow
- settings/policies

Potential later surfaces:
- search
- trends/insights
- sleep-focused view
- debugging / evidence view

### 1. Timeline

This is likely the home screen or at least the product center of gravity.

What it should do:
- show semantic segments over time
- show important overlays such as elevated heart rate, poor GPS quality, and later sleep
- make uncertainty visible without turning every row into a warning
- support filtering
- support drilling into a segment or collection

What it should not do:
- show raw samples by default
- overload the user with every minor event

Possible feel:
- chronological feed
- visually grouped by day
- strong emphasis on start/end time, type, confidence state when needed, and quick summary stats

Possible timeline items:
- walking segment
- running segment
- vehicle segment
- water activity segment
- flight segment
- low-confidence unknown segment
- review-needed quality event
- collection card
- sleep segment later

**JF Note**
We will want the ability to filter/search the timeline. For example, filter by activity type or search for a word in the title or a general location/area where the activity took place.
**End JF Note**

Updated conclusion:
- Filtering and search are core timeline capabilities, not optional later polish.
- The timeline should support activity-type filtering, review/quality filtering, title search, and location/area-based lookup.

### 2. Segment Detail

This is likely the main inspection and correction screen.

What it should show:
- map or route if relevant
- key stats
- visible class
- confidence and ambiguity when relevant
- pause events
- health overlays such as heart rate
- quality state
- notes and tags

What it should allow:
- reclassify
- retitle
- split
- trim
- delete
- add notes/tags
- add to collection
- export directly or via collection

Possible product principle:
- segment detail should explain the segment well enough that the user can confidently decide whether it is correct

### 3. Review Queue

This surface exists because ambiguity and bad data are part of the product.

What it should contain:
- low-confidence segments
- suspicious quality intervals
- ambiguous mixed segments
- items that need user confirmation

What it should optimize for:
- fast triage
- quick confirm/fix/dismiss actions
- minimal interruption during normal use

Possible principle:
- issues should accumulate quietly and be reviewable later unless the user explicitly wants more real-time prompts

### 4. Collection View

Collections are not just tags; they are a user-facing storytelling and organization layer.

What a collection should show:
- title
- cover map or summary visual
- included segments
- combined route where relevant
- combined stats
- notes/media
- export/share actions

What it should allow:
- rename
- reorder
- add/remove segments
- include notes/media
- create sub-collections

Possible examples:
- Saturday Ride
- Travel Day
- Triathlon
- Weekend Adventure

### 5. Export Flow

Export should feel like publishing, not like dumping raw data.

What the flow may need:
- choose source: segment or collection
- choose export style
- choose destination
- preview result
- optionally save export as an artifact

Possible export styles:
- route card
- stat summary
- story timeline
- animated route playback
- archive package

Possible destinations:
- Messages
- WhatsApp
- Email
- Instagram
- Strava-style integration later

**JF Note**
- Facebook
- TikTok
**End JF Note**

Updated conclusion:
- Export destinations should assume a broad sharing surface from the start, not just messaging apps and email.

### 6. Settings / Policies

Because this app is passive and resource-aware, settings matter more than in many apps.

Likely areas:
- battery policy
- low power mode behavior
- connectivity/upload behavior
- review behavior
- notification behavior
- retention preferences
- privacy/storage destination choices later

Important UI principle:
- settings should support user control without forcing constant configuration

## Likely Navigation Shape

At a conceptual level, the navigation may want to center around:
- Timeline
- Review
- Collections
- Settings

With Segment Detail and Export as contextual flows from those surfaces.

Possible interpretation:
- Timeline is the default home
- Review is the "attention needed" inbox
- Collections is the "curated history" layer
- Settings is operational control

### Alternative Navigation Thought

It may also be worth considering whether Review should be more of a badge or mode than a primary tab, depending on how often it is expected to matter.

**JF Note**
Instead of a dedicated Review list, we could filter the timeline by "Needs review", "Low confidence", etc.
**End JF Note**

Updated conclusion:
- Review does not need to be a primary standalone area by default.
- The simpler starting model is likely timeline-first with strong review filters and visible review tags on cards.

### UI Tone

The UI tone should probably balance:
- calm and archival
- trustworthy
- spatially rich when movement is involved
- explicit but not alarmist when uncertainty exists

It should probably not feel:
- gamified
- aggressively fitness-centric
- overloaded with engineering detail

## Product Behaviors The UI Must Support

The UI must support these behaviors cleanly:
- passive "just recorded it" confidence
- fast correction after the fact
- grouping segments into meaningful collections
- seeing quality issues without drowning in them
- seeing health overlays in context
- preparing something polished enough to share

## Tensions The UI Needs To Resolve

### 1. Richness vs Clarity

There is a lot of possible information:
- route
- speed
- elevation
- heart rate
- confidence
- quality
- notes
- pause events
- collection grouping

The UI will need strong hierarchy so that the default view is digestible.

### 2. Honesty vs Noise

You want ambiguity surfaced, but not every row should scream uncertainty.

This suggests:
- quiet defaults
- stronger visual treatment only when confidence or quality actually needs attention

### 3. Passive Logging vs Manual Curation

The app should work passively, but still make curation pleasant when you want it.

This suggests:
- editing tools should exist, but not dominate the primary surfaces

### 4. Segment Truth vs Collection Meaning

The UI must make it clear that:
- a segment is what the system detected
- a collection is how the user chooses to organize/share meaning

If this is blurred, the product may become confusing.

## UI Resting State

This branch now has a clearer direction.

### Recommended Main UI Structure

- Timeline as the default home
- Timeline built from adaptive cards
- Collections shown as first-class cards in the same feed
- Segment and Collection detail views as the main drill-down surfaces
- Review handled primarily through filters, tags, and timeline states rather than a dedicated default inbox
- Export as a quick-share flow with light customization
- Settings/policies as an important but secondary control surface

### Recommended Timeline Behavior

- group by day
- support filtering and search
- show segments, collections, and important event overlays
- visually tag low-confidence and needs-review items
- keep raw observations out of the default feed

### Recommended Detail Behavior

- map-forward for movement-heavy content
- clear stats and summaries
- visible quality and confidence context when relevant
- easy access to split, trim, retitle, reclassify, note, collect, and export actions

### Recommended Collection Behavior

- collections live on the main timeline
- collections suppress their child segments from the top-level timeline by default
- collection detail reveals the underlying segments
- collections support notes/media and sharing/export

### Recommended Watch Direction

- the watch app should show contextually appropriate UI for the current inferred activity
- it should provide an easy way to correct the detected activity
- it should prioritize glanceability and quick intervention over deep browsing

### Sensitive Activity Principle

If the product ever supports intimate or highly sensitive activity classes:
- detection should be opt-in
- storage should be opt-in
- timeline visibility should be separately opt-in
- exports should never include such content by accident
